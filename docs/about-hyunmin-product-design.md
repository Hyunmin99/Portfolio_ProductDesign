## One-line positioning

정량 데이터와 현장 맥락을 함께 의심하면서, 문제 정의부터 AI/CRM UX 설계, 프로토타입, 핸드오프, 측정 회수까지 연결하는 B2B SaaS 프로덕트 디자이너.

## Portfolio narrative to strengthen

### 1. 숫자로 결정하되, 숫자를 한 번 더 의심한다

최근 작업 로그에서 가장 강하게 반복된 패턴은 “데이터 기반 의사결정” 자체보다, **그 데이터가 정말 사용자 행동을 대표하는지 다시 검산하는 태도**다.

- 내부/테스트 사용량을 먼저 분리하고, 외부 고객 신호만 따로 보려 한다.
- 이벤트명이 실제 행동을 제대로 가리키는지 확인하고, 신·구 이벤트가 같은 행동을 중복 집계하지 않는지 점검한다.
- 단순 집계가 아니라 퍼널, 세그먼트, 사용 맥락, 인터뷰를 함께 보며 “많이 쓰는 것”과 “그 기능을 원하는 것”을 구분한다.
- 측정 결과가 애매할 때는 실패로 단정하지 않고, 표본 수·측정창·이벤트 설계 한계를 함께 기록한다.

**Portfolio copy**

> 저는 숫자를 근거로 삼지만, 숫자 하나만 믿지는 않습니다. 내부 사용량과 고객 사용량을 분리하고, 이벤트 정의와 측정 기준이 결과를 왜곡하지 않는지 먼저 확인합니다. 정량 데이터와 인터뷰·제품 맥락을 교차검증해 진짜 문제를 좁히는 방식으로 일합니다.

### 2. AI 기능을 ‘멋진 화면’보다 스코프·신뢰·상태 모델로 다룬다

AI 브리핑, CRM AI 어시스턴트, 이메일/녹음 관련 기능에서 공통적으로 드러나는 강점은 AI를 시각적으로 포장하는 것이 아니라 **무엇을 1차 범위로 만들고, 무엇을 후속 비전으로 남길지 판단하는 능력**이다.

- AI가 아직 정확히 판단할 수 없는 영역은 과감히 덜어내고, 현재 단계에서 신뢰 가능한 사실 정보로 대체한다.
- 즐겨찾기, 타임라인, 주간 활동처럼 사용자에게 바로 설명 가능한 모듈부터 1차 스코프로 재정렬한다.
- “AI가 추천한다”는 인상을 주기 전에, 추천의 근거·상태·실패 가능성을 먼저 따진다.
- AI assistant의 대상 오브젝트가 확장될 때, 카피와 예시 질문이 실제 제품 모델을 오해시키지 않도록 정보 층위를 정리한다.

**Portfolio copy**

> AI 기능을 설계할 때는 ‘AI답게 보이는 화면’보다 사용자가 어디까지 믿어도 되는지를 먼저 봅니다. 1차 스코프와 비전을 분리하고, 아직 지능이 붙지 않은 판단은 사실 정보·상태값·사용자 제어로 풀어 신뢰를 지키는 방향을 선택합니다.

### 3. 프로토타입과 자동화로 반복 케이스를 검증한다

최근 로그에서는 HTML 프로토타입, Figma 자동 주입, 스크린샷 검증, 케이스 매트릭스 정리처럼 **디자인 산출물을 빠르게 만들고 검증하는 운영 능력**이 자주 보인다.

- 알림처럼 케이스가 많은 기능은 먼저 렌더 코드에서 실제 문구·트리거·아이콘을 전수 조사한 뒤, 프로토타입과 Figma 케이스로 옮긴다.
- 반복 UI는 수작업으로 하나씩 복사하지 않고, 데이터 구조를 먼저 고정한 다음 자동화 가능한 형태로 주입한다.
- 프로토타입은 “예쁜 시안”이 아니라 케이스 전환, 상태 비교, 모션 타이밍, 접근성 대안까지 검토하는 도구로 사용한다.
- AI 도구를 디자인 보조로 쓰더라도, 마지막에는 DS 토큰, 타이포 스타일, 실제 Figma 노드 상태까지 확인한다.

**Portfolio copy**

> 반복 케이스가 많은 B2B 화면은 감으로 한두 개 그리지 않습니다. 실제 코드·문구·상태를 먼저 전수 조사하고, 프로토타입과 Figma 자동화로 케이스를 빠르게 펼친 뒤, 디자인 시스템 정합성과 예외 상태를 다시 검증합니다.

### 4. 카피를 ‘문장’이 아니라 실제 동작의 계약으로 본다

작업 로그에서 카피 관련 판단은 톤앤매너보다 **사용자가 어떤 결과를 기대하게 되는가**에 초점이 맞춰져 있다.

- 버튼명이 실제 동작과 다르면, 예쁘게 다듬기보다 먼저 의미를 바로잡는다.
- 되돌릴 수 없는 행동은 위험을 툴팁 뒤에 숨기지 않고 상시 노출한다.
- 특정 오브젝트에만 맞던 표현이 다른 오브젝트까지 확장될 때, 모든 범위에서 어색하지 않은 일반화된 표현으로 바꾼다.
- 고객에게 전달되는 문구는 상대의 결핍을 지적하지 않고, 필요한 맥락과 선택지를 제공하는 방식으로 정리한다.

**Portfolio copy**

> UX writing을 문장 다듬기로만 보지 않습니다. 버튼과 안내문은 사용자가 제품과 맺는 계약이라고 보고, 실제 시스템 동작·위험·복구 가능성을 문구에 정확히 반영하려고 합니다.

### 5. 핸드오프에서 끝내지 않고, 배포와 측정 회수를 붙인다

6월 회고의 중요한 자기진단은 “앞단의 문제 정의와 디자인은 강하지만, 배포 후 실제 사용자가 본 것과 지표 회수는 더 강하게 붙여야 한다”는 점이다. 이건 약점인 동시에 포트폴리오에서 오히려 신뢰를 줄 수 있는 성장 서사다.

- 검증 백로그를 따로 만들고, 놓친 재측정·측정 공백·개발 확인 사항을 분리한다.
- ‘디자인 완료’의 기준을 화면 산출물이 아니라 에러/충돌 상태, 배포본 확인, 측정 계획까지 확장하려 한다.
- 이벤트 부재나 중복 트래킹처럼 측정 인프라 문제를 디자인 의사결정의 일부로 끌어온다.
- 성과가 없다고 단정하기 전에, 측정 설계 자체가 충분한지 먼저 본다.

**Portfolio copy**

> 산출물로 끝나는 디자인보다, 실제 배포본과 측정 회수까지 이어지는 디자인을 지향합니다. 최근에는 검증 백로그를 따로 운영하며 놓친 지표, 이벤트 공백, 개발 확인 사항을 분리해 후속 액션으로 바꾸고 있습니다.

## Resume-ready bullets

- B2B CRM/Sales SaaS에서 정량 데이터, 인터뷰, 제품 맥락을 교차검증해 문제를 정의하고 PRD·Linear·Figma 산출물로 연결했다.
- AI 브리핑과 CRM AI assistant 흐름에서 1차 스코프와 후속 비전을 분리하고, 신뢰 가능한 정보 구조·상태값·카피 원칙을 설계했다.
- 알림, 모달, AI toast 등 케이스가 많은 UI를 코드 전수조사, HTML 프로토타입, Figma 자동화로 전개해 핸드오프 정확도를 높였다.
- 이벤트 설계와 측정 공백을 디자인 검증의 일부로 다루며, 배포 후 재측정 가능성·표본 한계·데이터 품질 이슈를 함께 기록했다.
- UX writing을 실제 시스템 동작과 사용자 기대의 계약으로 보고, 위험·복구 가능성·오브젝트 범위가 정확히 드러나도록 카피를 정리했다.

## Suggested additions for the next PDF portfolio revision

### About section

현재 PDF의 자기소개가 산출물 중심이라면, 아래 키워드를 보강하면 최근 작업 방식이 더 선명해진다.

- **Data-skeptical designer**: 데이터를 쓰되, 데이터가 만들어진 기준부터 의심하는 사람
- **AI product UX**: AI를 화면 장식이 아니라 신뢰·스코프·상태 모델로 설계하는 사람
- **B2B SaaS systems thinking**: 오브젝트, 권한, 이벤트, 핸드오프, 측정을 같이 보는 사람
- **Prototype-driven handoff**: 프로토타입과 자동화로 케이스를 펼치고 검증하는 사람
- **Post-launch learning**: 디자인 완료 후 배포·측정·검증까지 회수하려는 사람

### Case study candidates

1. **AI Briefing Home**  
   1차 스코프 재정렬, 즐겨찾기 정보 구조, 타임라인 레이아웃 판단, AI 신뢰성 기준을 보여주기 좋다.

2. **CRM Notification Redesign**  
   실제 렌더 코드 기반 전수조사, 30개 안팎의 케이스 정리, Figma 자동화, DS 아이콘·카테고리 체계를 보여주기 좋다.

3. **CRM AI Assistant / Sailo Expansion**  
   AI assistant가 다룰 수 있는 오브젝트 범위를 카피와 예시 질문으로 어떻게 설명했는지 보여주기 좋다.

4. **Recording / Mobile Usage Measurement**  
   기능 사용량 분석, 이벤트명 검산, 내부 데이터 제외, underpowered 측정 판단처럼 “디자인 리서치와 분석의 엄밀함”을 보여주기 좋다.

5. **Duplicate Data Modal / Risk Communication**  
   사용자가 위험을 모르고 선택하지 않도록 버튼 위계, 경고 노출, 복구 가능성을 어떻게 설계했는지 보여주기 좋다.

## Public-safety checklist before publishing detailed cases

- 고객사명, 사용자명, 내부 워크스페이스명은 익명화한다.
- Notion, Linear, Slack, DB, Mixpanel의 원문 링크나 화면 캡처는 공개하지 않는다.
- 정확한 내부 이벤트명과 raw metric은 필요할 때만 추상화해 사용한다.
- 수치가 필요한 경우에는 범위·비율·상대 변화 중심으로 재서술하고, 표본 한계도 함께 적는다.
- 내부 제품 전략이나 아직 출시 전인 스코프는 “concept / internal exploration”으로 분리한다.

## Compact profile draft

B2B CRM/Sales SaaS에서 문제 정의부터 AI UX, 프로토타입, 핸드오프, 측정 회수까지 연결하는 프로덕트 디자이너입니다. 저는 데이터를 근거로 삼되 데이터가 만들어진 기준을 먼저 의심하고, 내부 테스트와 실제 고객 행동을 분리해 문제를 좁힙니다. AI 기능을 설계할 때는 화려한 화면보다 사용자가 어디까지 믿어도 되는지, 어떤 상태와 예외를 드러내야 하는지, 무엇을 1차 스코프로 남겨야 하는지를 먼저 봅니다. 반복 케이스가 많은 B2B 화면은 코드 전수조사와 프로토타입, Figma 자동화를 활용해 빠르게 펼치고, 배포 후에도 이벤트·측정 공백·검증 백로그까지 챙기는 방향으로 일합니다.
